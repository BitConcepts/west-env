# Architecture — west-env

## Overview
`west-env` is a cross-platform Zephyr RTOS developer environment manager. It provides a unified `west env` CLI that adapts to the developer's host OS and available container/VM backend to deliver reproducible, performant Zephyr build environments on Windows, Linux, and macOS.

**Design axiom**: Windows must feel native. Builds run in Linux containers or VMs; the developer edits, flashes, and debugs from Windows without WSL, Remote SSH, or slow bind-mounts.

- **Languages**: Python 3.10+, YAML, PowerShell (wrappers), shell (wrappers)
- **Build system**: pyproject / setuptools
- **Test framework**: pytest (unit), GitHub Actions (integration), manual (platform/hardware)
- **Entry point**: `west env` (west extension registered via `[project.entry-points."west.commands"]`)

## Architecture Layers

### Layer 1 — Host UX Layer
Platform-native CLI entrypoints and wrapper scripts. PowerShell wrappers (`.ps1`) on Windows; shell wrappers (`.sh`) on Linux/macOS. Wrappers are generated by `west env generate-tasks`. Common command semantics are preserved across platforms so a developer workflow documented for one OS works with minimal change on another. (see REQ-PLATFORM-001, REQ-PLATFORM-002, REQ-PLATFORM-003)

### Layer 2 — Backend Abstraction Layer
Detects and selects the appropriate container/VM backend for the host. Exposes a uniform `backend.run(command, workspace_mode)` interface upstream. Reports the selection and rationale via `west env doctor`. (see REQ-BACKEND-001, REQ-BACKEND-002, REQ-BACKEND-003, REQ-BACKEND-004)

Platform-specific backend selection order:
- **Windows**: Podman machine (Hyper-V VM) → Docker Desktop (WSL2, fallback with warning) → host bind mount (not recommended)
- **Linux**: Podman native → Docker native
- **macOS**: Podman machine → Docker Desktop

### Layer 3 — Container Execution Layer
Runs commands inside the selected backend with correct shell-quoting and environment preparation. Injects `git config --global safe.directory '*'` before any command so Zephyr west extensions load correctly under Git ≥ 2.35. Delegates workspace volume/path strategy to Layer 4. (see REQ-CONTAINER-002, REQ-CONTAINER-003)

### Layer 4 — Workspace Synchronization Layer
Manages how source files and build artifacts move between host and container/VM. (see REQ-WORKSPACE-001, REQ-WORKSPACE-002, REQ-WORKSPACE-003, REQ-WORKSPACE-004)

Workspace modes:
- `sync` — rsync host → VM ext4 volume; sync build artifacts back. **Windows default.**
- `copy` — copy-in at build start, copy-out at end. Portable fallback.
- `tmpfs` — source synced, build directory in container tmpfs. Maximum speed.
- `bind` — direct host path bind mount. Linux/macOS only; emits warning on Windows.

Bind mount on Windows (`C:\...`) is allowed for compatibility but is never the default and always emits a performance warning. (see REQ-WINPERF-001, REQ-WINPERF-002, REQ-WINPERF-003, REQ-WINPERF-004)

### Layer 5 — Cache Layer
Persistent named volumes for ccache, west module cache, Zephyr SDK, and pip. Cache volumes survive container rebuilds. `west env cache stats` and `west env cache reset` are provided. (see REQ-CACHE-001, REQ-CACHE-002, REQ-CACHE-003, REQ-CACHE-004)

### Layer 6 — Git Credential Layer
Forwards host Git credentials into the container without copying tokens into images. On Windows: forwards the Windows OpenSSH agent socket (`SSH_AUTH_SOCK`). Falls back to HTTPS credential-manager delegation. (see REQ-GIT-001, REQ-GIT-002, REQ-GIT-003, REQ-GIT-004)

### Layer 7 — Flash / Debug Layer
On Windows, SEGGER J-Link for Windows is the default flash/debug path. Build artifacts are synced to the host (via Layer 4) and flashed with Windows J-Link tools. USB passthrough into the container is not required by default. An optional TCP J-Link/GDB server mode is provided for container-side debugging. (see REQ-JLINK-001, REQ-JLINK-002, REQ-JLINK-003, REQ-JLINK-004)

### Layer 8 — VSCode Integration Layer
Generates `.vscode/tasks.json` with tasks for: setup, sync, build, flash, debug, benchmark. Windows tasks call PowerShell wrappers; Linux/macOS tasks call shell wrappers. No Remote WSL task is generated or assumed. (see REQ-VSCODE-001, REQ-VSCODE-002, REQ-VSCODE-003, REQ-VSCODE-004)

## Platform Models

### Windows
- **Host tools**: Windows VSCode (no Remote WSL), PowerShell, J-Link for Windows, Windows OpenSSH
- **Recommended backend**: Podman machine (Hyper-V VM with ext4 named volume)
- **Fallback backend**: Docker Desktop (WSL2 bind mount — slow; warning emitted)
- **Workspace**: ext4 volume inside Podman/Hyper-V VM (no NTFS→Linux bind-mount overhead)
- **Flash**: Windows J-Link host tools on synced build artifact path
- **VSCode UX**: Windows tasks call `.ps1` wrappers — no Remote WSL extension needed
(see REQ-WINNUX-001, REQ-WINNUX-002, REQ-WINNUX-003, REQ-WINNUX-004)

### Linux
- **Host tools**: native shell, J-Link for Linux (optional), native OpenSSH
- **Recommended backend**: Podman or Docker native
- **Workspace**: host bind mount or named volume (bind mount is fine on Linux)
- **Flash**: Linux J-Link or USB passthrough into container

### macOS
- **Host tools**: native shell, Podman/Docker machine
- **Recommended backend**: Podman machine or Docker Desktop
- **Workspace**: named volume or copy mode (avoids macOS VirtioFS overhead on large trees)
- **Flash**: macOS J-Link host tools on synced artifacts

## Backend Models

- `podman-machine-hyperv` — Podman machine backed by Hyper-V VM. **Preferred on Windows.**
- `docker-desktop` — Docker Desktop (WSL2 backend). Fallback on Windows; bind mount warning.
- `podman-native` — Podman binary directly. Preferred on Linux.
- `docker-native` — Docker binary directly. Linux.
- `podman-machine` — Podman machine (non-Hyper-V). macOS preferred.
- `docker-machine` — Docker Desktop machine. macOS fallback.

Detection is performed by `west env doctor` which explains which backend was selected and why (see REQ-BACKEND-003).

## Configuration (west-env.yml)
Resolved from the manifest directory via `.west/config` (see REQ-CONFIG-001). Key top-level fields:

```yaml
env:
  backend: auto            # auto | podman-machine-hyperv | docker-desktop | podman-native | ...
  workspace_mode: sync     # sync | copy | tmpfs | bind
  image: ghcr.io/...
cache:
  ccache: true
  modules: true
git:
  credential_helper: auto  # auto | openssh-agent | credential-manager | none
jlink:
  mode: host               # host | tcp-server | none
```

`EnvConfig` parses `backend` and `workspace_mode` (see REQ-CONFIG-002, REQ-CONFIG-003, REQ-CONFIG-004).

## Internal Modules

### west_commands.env *(entry point — Slice 4–8 expansion target)*
Current: `init`/`build`/`shell`/`doctor` sub-actions. Target: expands to `sync`, `flash`, `cache`, `generate-tasks`. Pre-flight workspace validation retained. (see REQ-CMD-006)

### west_env.config *(Slice 2 expansion target)*
Parses `west-env.yml` from the manifest directory. Target: adds `backend`, `workspace_mode`, `cache`, `git`, `jlink` sections. (see REQ-CONFIG-001, REQ-CONFIG-002, REQ-CONFIG-003, REQ-CONFIG-004)

### west_env.backend *(Slice 2 — replaces west_env.engine)*
New module replacing `west_env.engine`. Detects all 6 backend types, implements fallback chain, and produces structured doctor output. (see REQ-BACKEND-001, REQ-BACKEND-002, REQ-BACKEND-003, REQ-BACKEND-004)

### west_env.container *(Slice 3–4 expansion target)*
Current: bind-mount execution with shell-quoting and git safe.directory. Target: delegates workspace strategy to `west_env.sync`. (see REQ-CONTAINER-002, REQ-CONTAINER-003)

### west_env.sync *(Slice 3 — new module)*
Implements `sync`, `copy`, `tmpfs`, `bind` workspace modes. Handles source-only sync and artifact sync-back with configurable exclusion patterns. (see REQ-WORKSPACE-001, REQ-WORKSPACE-002, REQ-WORKSPACE-003, REQ-WORKSPACE-004)

### west_env.cache *(Slice 5 — new module)*
Manages ccache, west module, Zephyr SDK, and pip named volumes. Provides `stats` and `reset` sub-commands. (see REQ-CACHE-001, REQ-CACHE-002, REQ-CACHE-003, REQ-CACHE-004)

### west_env.credentials *(Slice 9 — new module)*
Forwards host Git credentials (SSH agent socket or HTTPS credential-manager) into containers without copying tokens. (see REQ-GIT-001, REQ-GIT-002, REQ-GIT-003, REQ-GIT-004)

### west_env.flash *(Slice 10 — new module)*
Manages J-Link host flashing strategy and optional TCP GDB server mode. (see REQ-JLINK-001, REQ-JLINK-002, REQ-JLINK-003, REQ-JLINK-004)

### west_env.vscode *(Slice 8 — new module)*
Generates `.vscode/tasks.json` per platform. Windows tasks invoke `.ps1` wrappers; Linux/macOS invoke `.sh`. (see REQ-VSCODE-001, REQ-VSCODE-002, REQ-VSCODE-003, REQ-VSCODE-004)

### west_env.platform *(Slice 6–7 — new module)*
Generates and manages platform-specific wrapper scripts. (see REQ-PLATFORM-001, REQ-PLATFORM-002, REQ-PLATFORM-003)

### west_env.util *(stable)*
Python version check and host platform utilities. (see REQ-UTIL-001)

## Key Design Decisions

- **Windows-first workspace**: Default uses Podman Hyper-V VM with an ext4 named volume. Source files are rsynced into the VM; build artifacts sync back. Eliminates NTFS→Linux bind-mount overhead.
- **No Remote WSL**: Developers use standard Windows VSCode with generated tasks invoking PowerShell wrappers. No Remote WSL extension, no editing inside WSL.
- **J-Link on Windows host**: Flash/debug uses Windows-native J-Link tools on synced artifacts. USB passthrough into the container is not the default path.
- **Git credentials never copied**: Forwarded via SSH agent socket or OS credential manager; no tokens enter Docker images.
- **Backend selection is auditable**: `west env doctor` explains which backend was selected, which were detected, and which fallbacks were skipped.
- **Bind mount is a fallback on Windows**: Allowed for compatibility but emits a warning and is never the default.
- **Testability is a requirement**: Every requirement maps to at least one test entry; manual tests have exact steps. (see REQ-TESTABILITY-001, REQ-TESTABILITY-002, REQ-TESTABILITY-003)
